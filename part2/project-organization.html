6  Project Organization

Imagine opening a project you haven’t touched in six months. Or inheriting a colleague’s analysis when they rotate out of the lab. You need to figure out: where’s the raw data? Which script produces which figure? What’s the current version of the analysis? Is that CSV file something I downloaded, or something a script generated?

A well-organized project answers these questions through its structure alone. You shouldn’t need to read code or ask someone — the folder layout and naming conventions should tell you what’s what. This chapter describes the conventions we use in the Horsley Lab to make that possible.

In Chapter 3, you created a project using /new-project. This chapter explains that structure formally and covers how it scales as projects grow.

6.1 The Two Rules

Every analysis project has two types of files, and the entire organizational system follows from keeping them separate:

  1. Inputs are sacred. Data that comes from outside the project — sequencing results, collaborator files, public datasets — goes in data/ and is never modified by your scripts. If you need a cleaned version, your script reads the original, transforms it, and saves the result as an output.

  2. Outputs are disposable. Everything your scripts produce — processed Seurat objects, figures, tables, rendered reports — goes in outputs/ and can always be regenerated by rerunning the script that created it.

This separation is the foundation. When you know that everything in data/ came from outside and everything in outputs/ was generated by code, you can always trace where a file came from and how to recreate it.

6.2 Anatomy of a Lab Project

Here’s what a typical Horsley Lab project looks like:

my-project/
├── .claude/                  # Claude Code project config
│   └── CLAUDE.md
├── data/                     # External inputs — scripts never write here
│   ├── raw/                  # Original files, never modified
│   └── processed/            # Cleaned/transformed data
├── scripts/                  # R analysis scripts (.Rmd)
│   └── scratch/              # Exploratory, one-off analyses
├── outputs/                  # All generated outputs
│   ├── figures/
│   └── tables/
├── docs/                     # Notes, protocols, manuscript drafts
├── renv.lock                 # R package versions
├── .gitignore
└── README.md

The /new-project skill creates this entire structure for you — creating directories, initializing git, setting up renv, and generating a .claude/CLAUDE.md with your project’s conventions. But it’s worth understanding what each piece does.

6.3 data/ — Your Inputs Are Sacred

The data/ folder holds files that come from outside the project — things your scripts read but never write to:

data/
├── raw/
│   ├── counts_matrix.h5          # From the sequencing core
│   ├── sample_metadata.xlsx      # From a collaborator
│   └── README.md                 # Documents where each file came from
└── processed/
    ├── seu_filtered.rds          # Seurat object after QC
    └── metadata_annotated.csv    # Cleaned metadata

The raw/ subdirectory holds truly original files — Cell Ranger output, sequencing data from the core, metadata spreadsheets from collaborators, reference files downloaded from databases. Scripts never write to data/raw/.

The processed/ subdirectory is for cleaned or transformed data that you trust enough to use as inputs in later scripts — for example, a filtered Seurat object that took 30 minutes to generate, or a metadata table you spent time cleaning by hand. These are still inputs, not outputs, so they live in data/ rather than outputs/.

Document where each file came from. A data/raw/README.md is the simplest approach — note the source, date received, and any relevant details for each file. Six months from now, you’ll be grateful you wrote down which version of the genome reference you used, or which email attachment that metadata came from.

6.4 scripts/ — Numbered Analysis Scripts

All analysis scripts live in scripts/, numbered to show their logical flow:

scripts/
├── 01_qc.Rmd
├── 02_normalization.Rmd
├── 03_clustering.Rmd
├── 04_annotation.Rmd
├── 05_de_analysis.Rmd
└── scratch/
    └── test_resolution_params.Rmd

6.4.1 Why .Rmd Files?

In the lab, all data analysis scripts are R Markdown documents (.Rmd), not plain .R scripts. R Markdown lets you combine code, results, and narrative explanation in a single file — so your analysis documents what it does and why as it runs. When you knit an .Rmd file, it produces an HTML report with your figures, tables, and text woven together.

Use plain .R files only for helper functions that you source() into your main scripts.

6.4.2 Why Numbers?

The two-digit prefix (01_, 02_, etc.) serves a simple purpose: when you look at the file explorer, scripts appear in the order of your analysis pipeline. Script 01 imports and does QC. Script 02 normalizes. Script 03 clusters. A new lab member can glance at the file list and understand the analysis flow.

The numbers indicate logical order, not strict dependencies. Script 05 might read outputs from scripts 01 and 03 — the numbering just helps you understand the overall structure at a glance.

When you add a new script, assign the next available number. If you archive or delete a script, don’t renumber the remaining ones — leave gaps. This avoids confusion with documentation that mentions the old numbers.

6.5 outputs/ — Provenance Through Structure

Here’s the convention that makes everything traceable: every script gets a matching output folder. Script 03_clustering.Rmd writes all its outputs to outputs/03_clustering/:

outputs/
├── figures/
│   ├── 03_clustering/
│   │   ├── umap_clusters.pdf
│   │   └── cluster_sizes.pdf
│   └── 05_de_analysis/
│       └── volcano_plot.pdf
├── tables/
│   ├── 03_clustering/
│   │   └── cluster_summary.csv
│   └── 05_de_analysis/
│       └── de_results.csv
└── 03_clustering/
    ├── seu_clustered.rds
    └── BUILD_INFO.txt

This structure encodes provenance automatically. When you see a file in outputs/03_clustering/, you know exactly which script created it. The Seurat object, figures, and tables from each script are clearly associated with that script.

Output ownership is strict: a script writes only to its own output folder, never to another script’s. If script 05 needs a modified version of something script 03 produced, it reads script 03’s output and saves its own version in outputs/05_de_analysis/.

6.5.1 BUILD_INFO.txt

Every numbered script writes a BUILD_INFO.txt to its output folder as its last action:

script: 03_clustering.Rmd
commit: a1b2c3d
date: 2024-03-20 15:30:00

This answers a question that comes up constantly: “When was this output last regenerated, and from what version of the code?” If downstream figures look wrong, you can check the upstream folder’s BUILD_INFO.txt to see whether the Seurat object was generated from current code or something stale.

Add this to the end of every numbered script:

# Write build info
build_info <- data.frame(
  script = "03_clustering.Rmd",
  commit = system("git rev-parse --short HEAD", intern = TRUE),
  date = as.character(Sys.time())
)
write.csv(build_info, file.path(dir_out, "BUILD_INFO.txt"), row.names = FALSE)

6.5.2 Setting Up Output Paths

In your setup chunk, define paths that match this structure:

library(tidyverse)
library(here)
library(Seurat)

# This script's output folder
dir_out <- here::here("outputs", "03_clustering")
dir.create(dir_out, recursive = TRUE, showWarnings = FALSE)
dir.create(here::here("outputs", "figures", "03_clustering"), 
           recursive = TRUE, showWarnings = FALSE)

# Input paths
path_seu <- here::here("outputs", "02_normalization", "seu_normalized.rds")

Then save all outputs to dir_out:

# Save Seurat object
saveRDS(seu, file.path(dir_out, "seu_clustered.rds"))

# Save figures
ggsave(
  here::here("outputs", "figures", "03_clustering", "umap_clusters.pdf"),
  p_umap, width = 6, height = 5
)

6.6 How Scripts Connect

Scripts read from two places: data/ (external inputs) and outputs/ (outputs from other scripts). Group all input reads at the top of each script, with comments distinguishing external data from other scripts’ outputs:

# --- Inputs (from other scripts) ---
seu <- readRDS(here("outputs/02_normalization/seu_normalized.rds"))

# --- Inputs (external data) ---
metadata <- read_csv(here("data/raw/sample_metadata.csv"))
marker_genes <- read_csv(here("data/raw/published_skin_markers.csv"))

Reading the top of any script shows exactly what it depends on and where those files come from.

6.7 The Scratch Directory

Not everything you write is part of the main analysis pipeline. Sometimes you need to test a clustering resolution, try a new visualization, or run a quick sanity check. That’s what scripts/scratch/ is for:

scripts/
├── 01_qc.Rmd
├── 02_normalization.Rmd
├── scratch/
│   ├── test_resolution_params.Rmd
│   └── compare_normalization_methods.Rmd

Scratch scripts follow relaxed rules:

  • No number prefixes or BUILD_INFO.txt required
  • No other script depends on them — this is the critical rule. Scratch scripts can read from any outputs/ folder, but nothing outside of scratch/ reads from scratch outputs. This is a one-way dependency.
  • Clean them out at the end of each session — use /done to consolidate useful scratch code into numbered scripts
TipWhen scratch becomes permanent

If a scratch analysis proves valuable, promote it to a numbered script in the main directory. Use /done at the end of each session — it will check scripts/scratch/ and help you consolidate useful code into the right numbered script.

6.8 Growing a Project: Flat vs. Sectioned

The structure shown above is a flat layout — all scripts in one directory, one numbering sequence. This works well for most projects.

When a project grows to include multiple distinct analyses — say, scRNAseq and spatial transcriptomics from the same experiment — a flat layout gets unwieldy. Switch to a sectioned layout:

project/
├── scripts/
│   ├── scrna/
│   │   ├── 01_qc.Rmd
│   │   ├── 02_normalization.Rmd
│   │   └── 03_clustering.Rmd
│   ├── spatial/
│   │   ├── 01_load_visium.Rmd
│   │   └── 02_spatial_de.Rmd
│   ├── combined/
│   │   └── 01_integration.Rmd
│   └── scratch/
├── data/
│   ├── scrna/
│   └── spatial/
└── outputs/
    ├── scrna/
    │   ├── 01_qc/
    │   └── 02_normalization/
    ├── spatial/
    │   └── 01_load_visium/
    └── combined/
        └── 01_integration/

A few things to notice:

  • Numbering restarts at 01_ in each section — each analytical thread has its own sequence
  • Data and outputs mirror the section structure — scripts/scrna/ has a corresponding data/scrna/ and outputs/scrna/
  • Cross-section scripts like combined/01_integration.Rmd can read from any section’s outputs/ folder

Start flat. Switch to sectioned when you have more than about ten scripts or two distinct analytical threads.

6.9 Script Lifecycle

Track the status of each script with a status field in the YAML frontmatter:

---
title: "03 — Clustering"
date: "`r Sys.Date()`"
status: development
output: html_document
---
Status Meaning
development In active development; outputs may change
finalized Outputs are publication-ready; modify only deliberately
deprecated Superseded by a newer script; kept for reference

When a script is superseded, move it to scripts/old/ and add a note in the YAML:

---
title: "Old Clustering Approach"
status: deprecated
deprecated_by: 03_clustering.Rmd
output: html_document
---

6.10 Naming Conventions

Good file names are lowercase, descriptive, and use underscores:

  • Scripts: 03_clustering.Rmd, not 3-Clustering.Rmd or clustering final.Rmd
  • Output files: seu_clustered.rds, not data.rds or output_final_v2.rds
  • Figures: umap_cell_types.pdf, not figure1.pdf

Avoid spaces (they cause problems in terminal commands), generic names (results.csv), and date prefixes on files — let git track versions instead.

6.11 Version Control

Your .gitignore should include:

# Raw data (often large)
data/raw/

# Generated outputs (reproducible from scripts)
outputs/

# R artifacts
.Rhistory
.RData
renv/library/
renv/staging/

# Quarto rendering
*_files/
.quarto/

# OS and IDE files
.DS_Store
.positron/

# Secrets
.env
*.pem
credentials.json

Whether to commit data/raw/ depends on file sizes. Small files (a few MB) can be committed so the project is self-contained. Large files (Cell Ranger output, raw FASTQ) should be gitignored, with a data/raw/README.md documenting where to get them.

See ?sec-git for a full introduction to version control.

6.12 A Complete Example

Here’s a well-organized wound healing scRNAseq project:

wound_healing_scrna/
├── .claude/
│   ├── CLAUDE.md
│   └── WOUND_HEALING_SCRNA_PLAN.md   # Planning doc
├── data/
│   ├── raw/
│   │   ├── README.md                 # Documents data sources
│   │   ├── WH_001/                   # Cell Ranger output, sample 1
│   │   ├── WH_002/                   # Cell Ranger output, sample 2
│   │   └── sample_metadata.csv       # Experimental design
│   └── processed/
│       └── seu_merged.rds            # Merged object (slow to generate)
├── scripts/
│   ├── 01_qc.Rmd                     # status: finalized
│   ├── 02_normalization.Rmd          # status: finalized
│   ├── 03_clustering.Rmd             # status: finalized
│   ├── 04_annotation.Rmd             # status: development
│   ├── 05_de_analysis.Rmd            # status: development
│   ├── scratch/
│   │   └── test_harmony_params.Rmd
│   └── old/
│       └── 03_clustering_v1.Rmd      # status: deprecated
├── outputs/
│   ├── figures/
│   │   ├── 01_qc/
│   │   │   ├── violin_nfeature.pdf
│   │   │   └── violin_pct_mt.pdf
│   │   └── 03_clustering/
│   │       └── umap_clusters.pdf
│   ├── tables/
│   │   └── 05_de_analysis/
│   │       └── de_results.csv
│   ├── 02_normalization/
│   │   ├── seu_normalized.rds
│   │   └── BUILD_INFO.txt
│   └── 03_clustering/
│       ├── seu_clustered.rds
│       └── BUILD_INFO.txt
├── docs/
│   └── cell_type_marker_notes.md
├── renv.lock
├── .gitignore
└── README.md

From this structure, anyone can understand the project without reading a single line of code:

  • Where did the data come from? Check data/raw/README.md
  • What’s the analysis pipeline? Read the numbered scripts in order
  • Which script produced seu_clustered.rds? It’s in outputs/03_clustering/, so script 03 made it
  • Is the analysis finalized? Check the status field — scripts 01–03 are finalized, 04–05 are still in development
  • What code version produced these outputs? Check BUILD_INFO.txt in each output folder
  • How do I reproduce everything? Restore packages with renv::restore(), then run the scripts in order
